# NAM56R MIMO 7/7 VPP PP=2 — 112,152 tok/sec (2026-04-11)

**First successful VPP run of the complete NAM56R stack with full Mamba3 MIMO R=4** on LOCATION_2 H200×8. Doubles the baseline throughput from 56k to 112k tok/sec via pipeline parallelism.

## Result

- **112,152 tok/sec** steady-state mean (range 110k-113k)
- **2,337 ms/iter** (range 2,319-2,382 ms, very stable)
- **2.0× speedup** over PP=1 baseline (56,280 tok/sec)
- **Target 110k tok/sec: HIT** at this configuration
- 30/30 iterations completed, loss 11.92 → 2.73, grad norm 2.8, no NaN

## Blockers found and workarounds

### 1. Megatron fused MLA RoPE + PP>1 = CRASH (upstream bug)

**Symptom:** `rotary_fwd_kv_kernel` (Triton kernel in `fused_mla_yarn_rope_apply.py`) crashes with `cudaErrorIllegalAddress` during Triton autotuner benchmarking when PP>1.

**Root cause:** The MLA internal RotaryEmbedding caches cos/sin tensors that are device-global. With PP>1, different pipeline stages have different tensor layouts and the cached tensors from stage 0 are incompatible with stage 1's memory layout.

**Workaround:** `--no-rope-fusion` — uses the unfused Python-level RoPE application instead of the Triton fused kernel. Performance cost is ~10-30 ms/iter (unfused RoPE is a few small elementwise ops).

**Action:** Report upstream to Megatron-core. The fused MLA RoPE Triton kernel should handle PP>1 by per-stage caching or invalidation.

### 2. MBS=8 + PP=2 = OOM (132/140 GB)

With PP=2 (26 layers per stage), MBS=8 at seq=4096 overflows H200's 140 GB HBM. Peak allocation reaches 132 GB before the first forward completes.

**Resolution:** MBS=4 is the sweet spot — 118 GB peak on rank 0, 94 GB on rank 4. Adequate headroom.

### 3. MBS=8 + PP=2 + VPP=2 + `--no-rope-fusion` — attempted, still OOM

Even with the RoPE fix, MBS=8 OOMs. This is a genuine memory constraint of the 26-layer-per-stage PP=2 configuration, not a bug.

## Memory footprint

| Rank | Role | Peak allocated | Peak reserved |
|---|---|---|---|
| 0 | PP stage 0 (has embedding) | 118 GB | ~125 GB |
| 4 | PP stage 1 (has output head) | 94 GB | ~100 GB |

H200 has 140 GB HBM3 — rank 0 is within ~22 GB of the ceiling. MBS=5 might fit (~125 GB estimate) but MBS=6+ is risky.

## Path to 250k tok/sec from this baseline

112k tok/sec = 2.23× gap to 250k. Remaining optimization levers:

| Lever | Expected gain | New projected tok/sec | Effort |
|---|---|---|---|
| VPP baseline (this run) | — | 112k | done |
| + FP8 on MLA+MoE (halve ~250 ms WGMMA GEMM) | +15-20% | ~130-135k | small |
| + Restore rope fusion (fix upstream bug) | +2-3% | ~135-140k | medium |
| + MBS=5 or 6 (if memory fits) | +10-15% | ~150-160k | small |
| + cuBLASLt algo caching | +5% | ~160-170k | trivial |
| + bf16 softmax | +2% | ~165-175k | small |
| + PP=4 VPP=4 MBS=2 GBS=64 (finer pipeline, less bubble) | +10-20% | ~180-200k | medium |
| + TP=2 (2-way tensor parallelism + PP=2) | +15-25% | ~200-230k | medium |
| **Aggressive best case** | — | **~200-230k** | — |

Full 250k likely requires TP=2 + PP=2 + VPP=2 + FP8 + MBS optimization.
