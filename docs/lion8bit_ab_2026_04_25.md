# Lion8bit Muon Scalar Fallback A/B - 2026-04-25

## Summary

Goal: evaluate whether cppmega should switch Muon's non-2D/scalar fallback from
`adam8bit` to `lion8bit`.

Recommendation: keep `adam8bit` as the default for now; use `lion8bit` only as
an explicit opt-in until a longer full-pattern run clears. After the M2RNN
checkpointed backward fix, the default `AEMEAEMEAEMR` pattern passed a matched
2-iteration GB10 A/B. Lion8bit saved the expected ~0.34 GiB and the short
loss/grad-norm curves matched closely, but two iterations are only a
stability/configuration smoke, not convergence evidence.

## Inspection

- Local Megatron accepts `--muon-scalar-optimizer` choices `adam`, `lion`,
  `adam8bit`, and `lion8bit`.
- Megatron validates 8-bit scalar fallbacks under Muon plus bf16 no-master
  fallback.
- Megatron optimizer construction has both `adam8bit` and `lion8bit` branches;
  `lion8bit` uses `bitsandbytes.optim.Lion8bit`.
- `bitsandbytes` imported from the cppmega venv as `0.49.2`; `bnb.optim.Lion8bit`
  exists.
- `scripts/local_gb10_quarter_train.sh` now defaults
  `CPPMEGA_MUON_SCALAR_OPTIMIZER="${CPPMEGA_MUON_SCALAR_OPTIMIZER:-adam8bit}"`,
  so Lion8bit can be tested without editing the launcher.

## Caveats

- The default-pattern A/B now exercises the `R`/M2RNN path, but it is still only
  2 iterations.
- GB10 `nvidia-smi` memory fields reported unsupported/blank peak output, so
  memory conclusions use PyTorch memory hooks and the launcher's configured
  state estimate.
- Two iterations are enough for import/configuration and obvious instability
  smoke testing, not convergence.

## Recommendation

Do not flip the default yet. Keep `adam8bit` as the launcher default and treat
`lion8bit` as opt-in until a longer run on the full default pattern passes. The
full-pattern A/B is encouraging: losses and grad norms were essentially matched,
and the expected ~0.34 GiB saving showed up. That is still not enough evidence
to make Lion's different update rule the default for all scalar fallback
parameters.
